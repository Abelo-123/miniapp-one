The Chapa payment integration handles DEPOSITS (adding money) and WITHDRAWALS 
(removing money) for the betting app's internal wallet system.

                                              ┌─────▼─────┐
                                              │  MySQL DB  │
                                              │  (betting  │
                                              │   _site)   │
                                              └───────────┘

PAYMENT MODES:
  Mode 1: INLINE SDK (Primary) — Chapa's JavaScript SDK renders a payment form
           directly inside the wallet page. No redirect needed.

           redirected to Chapa's hosted checkout page.

3. DATABASE SCHEMA (Payment-Related Tables)
================================================================================

─────────────────── TABLE: users ───────────────────
Relevant columns for payment:
  id              INT UNSIGNED AUTO_INCREMENT PRIMARY KEY
  email           VARCHAR(255) UNIQUE NOT NULL
  full_name       VARCHAR(100) NOT NULL
  balance         DECIMAL(12,2) DEFAULT 0.00    ← The user's wallet balance
  role            ENUM('user','admin') DEFAULT 'user'


─────────────────── TABLE: deposits ───────────────────
Tracks every deposit attempt.

  id              INT UNSIGNED AUTO_INCREMENT PRIMARY KEY
  user_id         INT UNSIGNED NOT NULL         ← FK → users(id)
  amount          DECIMAL(12,2) NOT NULL        ← Amount in ETB
  tx_ref          VARCHAR(100) UNIQUE NOT NULL  ← Our unique reference (DEP-{userId}-{time}-{hex})
  chapa_tx_ref    VARCHAR(100) NULL             ← Chapa's own reference ID
  checkout_url    VARCHAR(500) NULL             ← Chapa's hosted checkout URL (redirect mode)
  status          ENUM('pending','success','failed','expired') DEFAULT 'pending'
  chapa_response  JSON NULL                     ← Full Chapa API response stored as JSON
  created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
  completed_at    DATETIME NULL                 ← When payment was confirmed

  INDEXES: idx_user(user_id), idx_tx_ref(tx_ref), idx_status(status)


─────────────────── TABLE: withdrawals ───────────────────
Tracks manual withdrawal requests (admin-approved).

  id              INT UNSIGNED AUTO_INCREMENT PRIMARY KEY
  user_id         INT UNSIGNED NOT NULL         ← FK → users(id)
  amount          DECIMAL(12,2) NOT NULL
  bank_name       VARCHAR(100) NOT NULL         ← e.g., "Commercial Bank of Ethiopia"
  account_number  VARCHAR(50) NOT NULL
  account_holder  VARCHAR(100) NOT NULL
  status          ENUM('pending','processing','approved','rejected') DEFAULT 'pending'
  admin_note      TEXT NULL
  processed_by    INT UNSIGNED NULL             ← FK → users(id) (admin who processed)
  created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
  processed_at    DATETIME NULL

  INDEXES: idx_user(user_id), idx_status(status)


─────────────────── TABLE: transactions ───────────────────
Immutable ledger of all money movements.

  id              INT UNSIGNED AUTO_INCREMENT PRIMARY KEY
  user_id         INT UNSIGNED NOT NULL
  type            ENUM('deposit','withdrawal','withdrawal_rejected','bet_placed',
                       'bet_won','bet_void','bonus') NOT NULL
  amount          DECIMAL(12,2) NOT NULL         ← Positive = credit, Negative = debit
  balance_after   DECIMAL(12,2) NOT NULL         ← Snapshot of balance after this transaction
  reference_type  VARCHAR(30) NULL               ← 'deposit', 'withdrawal', 'bet'
  reference_id    INT UNSIGNED NULL               ← FK to deposits.id or withdrawals.id or bets.id
  description     VARCHAR(255) NULL               ← Human-readable description
  created_at      DATETIME DEFAULT CURRENT_TIMESTAMP

  INDEXES: idx_user(user_id), idx_type(type), idx_created(created_at)

WHAT IT DOES:

─── CLASS: Chapa ───

  PROPERTIES:

    $baseUrl    (string) — Loaded from CHAPA_BASE_URL constant

  METHOD: __construct()
    Sets secretKey and baseUrl from constants.

  METHOD: initialize(array $data): array
    PURPOSE: Start a new payment transaction on Chapa.
    PARAMETERS (via $data array):
      - amount      (float)  — Amount in ETB
      - email       (string) — Customer email (fallback: 'customer@gmail.com')
      - first_name  (string) — Customer first name
      - last_name   (string) — Customer last name
      - tx_ref      (string) — Our unique transaction reference

    ALGORITHM:
      1. Validate email; if invalid, use fallback 'customer@gmail.com'
      2. Build payload:
         {
           amount, currency:'ETB', email, first_name, last_name, tx_ref,
           callback_url: SITE_URL + '/api/wallet/chapa_callback.php',
           return_url:   SITE_URL + '/pages/wallet.php?deposit=success',
           customization: { title: 'Wallet Deposit', description: 'Wallet deposit' }
         }
      3. Send POST to /transaction/initialize
      4. Return normalized result

    RETURNS:
      { success: bool, http_code: int, data: array, message: string, raw: array }
      On success, data.checkout_url contains the Chapa hosted checkout URL.

  METHOD: verify(string $txRef): array
    PURPOSE: Verify a transaction's status with Chapa.
    ALGORITHM:
      1. Send GET to /transaction/verify/{$txRef}
      2. Return normalized result
    RETURNS:
      { success: bool, http_code: int, data: array, message: string, raw: array }
      On success, data.status = 'success', data.amount, data.chapa_ref available.

  METHOD: request(string $method, string $endpoint, array $data): array  [PRIVATE]
    PURPOSE: Low-level cURL HTTP request wrapper.
    ALGORITHM:
      1. Initialize cURL to baseUrl + endpoint
      2. Set headers: Authorization: Bearer {secretKey}, Content-Type: application/json
      3. Set timeout: 30 seconds
      4. If POST: attach JSON-encoded payload
      5. Execute request
      6. Get HTTP status code
      7. Decode JSON response
      8. Return normalized array:
         success = (httpCode === 200 AND decoded.status === 'success')

    ALGORITHM:

      2. IF exists: return success with existing deposit_id (idempotent)

         b. Return { success: true, tx_ref, deposit_id }

  METHOD: verifyAndCompleteDeposit(string $txRef): array
  ================================================================
    PURPOSE: Verify a pending deposit with Chapa API and credit balance if confirmed.
    USED BY: api/wallet/verify_deposit.php (retry verification)

    ALGORITHM:
      1. BEGIN TRANSACTION
      2. SELECT deposit WHERE tx_ref = ? AND status = 'pending' FOR UPDATE (row lock)
      3. IF deposit not found:
         a. Check if already completed (status = 'success')
         b. IF already completed: return success with current balance
         c. ELSE: return error 'Deposit not found or already processed'
      4. Call Chapa verify($txRef)
      5. IF Chapa confirms (status = 'success'):
         a. Get verified amount from Chapa response
         b. UPDATE deposit: status='success', chapa_tx_ref, chapa_response, completed_at
         c. UPDATE users: balance = balance + verifiedAmount
         d. Get new balance
         e. INSERT transaction record (type='deposit')
         f. COMMIT
         g. Return { success: true, new_balance }
      6. IF Chapa not confirmed:
         a. COMMIT (leave as pending — don't mark failed so retry is possible)
         b. Return { success: false, message: 'Payment not yet confirmed' }
      7. ON EXCEPTION: ROLLBACK and return error

ALGORITHM:

     b. UPDATE deposit: status='success', chapa_tx_ref, chapa_response, completed_at

     d. Get new balance
     e. INSERT transaction record
     f. COMMIT
     g. Return { success: true, new_balance, verified: true }
  9. IF CHAPA NOT VERIFIED (or unreachable):
     a. COMMIT (leave as pending — callback will handle later)
     b. Return { success: true, pending: true, message: '...' }
  10. ON EXCEPTION: ROLLBACK + return error

9. DEPOSIT FLOW — STEP-BY-STEP ALGORITHMS
================================================================================

═══════════════════════════════════════════
FLOW A: INLINE SDK DEPOSIT (PRIMARY)
═══════════════════════════════════════════

This is the preferred flow when Chapa's JS SDK loads successfully.

STEP 1 — USER ENTERS AMOUNT
  User opens wallet page → clicks "Deposit" → enters amount → clicks Deposit button
  → initDeposit() calls startChapaPayment(amount)

STEP 2 — GENERATE TX_REF ON FRONTEND
  txRef = 'DEP-' + userId + '-' + Date.now() + '-' + random8chars
  Example: 'DEP-1-1709829000123-a1b2c3d4'

STEP 3 — CREATE DEPOSIT RECORD IN DATABASE
  Frontend →  POST /api/wallet/deposit.php
              Body: { amount: 100, tx_ref: 'DEP-1-...' }
  Backend  →  Wallet::createDepositRecord()
              INSERT INTO deposits (user_id, amount, tx_ref, status='pending')
  Response →  { success: true, tx_ref: '...', deposit_id: 42 }

STEP 4 — SHOW CHAPA INLINE FORM
  Hide the Deposit button
  Show the Chapa container
  Initialize ChapaCheckout:
    new ChapaCheckout({
      publicKey:    walletConfig.chapaPublicKey,
      amount:       amount (rounded to 2 decimals),
      currency:     'ETB',
      tx_ref:       txRef (same as stored in DB),
      email:        user's email,
      first_name:   user's name,
      callbackUrl:  /api/wallet/chapa_callback.php,
      ...
    })
  Call chapa.initialize() → renders payment form inside #chapa-inline-form

STEP 5 — PHONE VALIDATION (Frontend Enhancement)
  After 1 second, scan for phone input inside Chapa form
  Attach custom validator:
  - Telebirr numbers: start with 09 (9-digit base)
  - M-Pesa numbers: start with 07 (7-digit base)
  - Prevents submitting wrong number for selected payment method
  - Shows inline validation messages

STEP 6 — USER COMPLETES PAYMENT
  User enters phone → selects payment method → confirms payment
  Chapa SDK triggers onSuccessfulPayment callback

STEP 7 — CREDIT BALANCE (Backend Verification)
  Frontend calls completeDepositDirect(txRef, amount, chapaRef, sdkResult)
  → POST /api/wallet/complete_deposit.php
    Body: { tx_ref, amount, chapa_ref, sdk_response }

  Backend Algorithm (complete_deposit.php):
    1. Find deposit by tx_ref (FOR UPDATE lock)
    2. If not found, create one as pending
    3. If already completed, return success + balance (no double credit)
    4. VERIFY WITH CHAPA API:
       GET https://api.chapa.co/v1/transaction/verify/{tx_ref}
       Authorization: Bearer {SECRET_KEY}
    5. If Chapa confirms (status=200, data.status='success'):
       a. Use Chapa's VERIFIED amount (security: never trust frontend amount)
       b. UPDATE deposit status = 'success'
       c. UPDATE users balance += verified_amount
       d. INSERT transaction record
       e. Return { success: true, new_balance, verified: true }
    6. If Chapa not confirmed (network issue, delay):
       a. Leave deposit as pending
       b. Return { success: true, pending: true }
       c. Chapa callback will handle it later

STEP 8 — FRONTEND UPDATE
  On verified response:
    → Show "Balance updated!" toast
    → Update balance display elements
    → Reload page in 2 seconds
  On pending response:
    → Show "Balance will update shortly" toast
    → Reload page in 5 seconds
  On error:
    → Show positive fallback message ("Payment received!")
    → Reload page in 5 seconds (callback will catch it)


═══════════════════════════════════════════
FLOW B: REDIRECT DEPOSIT (FALLBACK)
═══════════════════════════════════════════

Used when Chapa's JS SDK fails to load (CDN down, browser blocks, etc.)

STEP 1 — USER CLICKS DEPOSIT
  SDK not available → isChapaAvailable() returns false
  → startRedirectPayment(amount) is called

STEP 2 — SERVER-SIDE INITIALIZATION
  Frontend → POST /api/wallet/deposit.php
             Body: { amount: 100 }  (NO tx_ref — server generates it)

  Backend (deposit.php → Wallet::createDeposit()):
    1. Generate tx_ref: 'DEP-{userId}-{time}-{random_hex}'
    2. Fetch user email and name from DB
    3. INSERT pending deposit into `deposits` table
    4. Call Chapa::initialize() with amount, email, name, tx_ref
    5. Chapa returns checkout_url
    6. Store checkout_url in deposit record
    7. Return { success: true, checkout_url: 'https://checkout.chapa.co/...' }

STEP 3 — REDIRECT TO CHAPA
  Frontend receives checkout_url → window.location.href = checkout_url
  User is now on Chapa's hosted checkout page.

STEP 4 — USER PAYS ON CHAPA'S PAGE
  User enters phone, selects method, completes payment.
  
STEP 5 — CHAPA CALLBACKS

  5A: RETURN URL (user's browser)
    After payment, user is redirected back to:
    wallet.php?deposit=success
    → Page loads normally, showing updated balance

  5B: CALLBACK URL (server-to-server)
    Chapa's server calls:
    /api/wallet/chapa_callback.php?tx_ref=DEP-1-...
    → handleChapaCallback() verifies with Chapa API
    → Credits user balance
    → Redirects to wallet.php?deposit=success


═══════════════════════════════════════════
FLOW C: ERROR RECOVERY / SAFETY NET
═══════════════════════════════════════════

Multiple layers ensure no payment is lost:

LAYER 1: complete_deposit.php verifies with Chapa on SDK success
LAYER 2: If verify fails, deposit stays 'pending', callback handles it later
LAYER 3: chapa_callback.php is called by Chapa's server (independent of browser)
LAYER 4: verifyDepositWithRetry() polls 4 times with increasing delays
LAYER 5: On any ambiguity, positive message shown + page reload

ANTI DOUBLE-CREDIT MECHANISM:
  - Every credit operation checks deposit.status first
  - FOR UPDATE row locking prevents race conditions
  - If status is already 'success', return balance without crediting again
  - Transaction is atomic (BEGIN...COMMIT/ROLLBACK)

10. WITHDRAWAL FLOW — STEP-BY-STEP ALGORITHM
================================================================================

NOTE: Withdrawals are MANUAL — they are processed by an admin, not via Chapa API.
Chapa's API is used only for deposits (incoming payments).

STEP 1 — USER SUBMITS REQUEST
  User clicks "Withdraw" → modal opens → fills form:
    - Amount (minimum 100 ETB)
    - Bank Name
    - Account Number
    - Account Holder Name
  Clicks "Submit"

STEP 2 — FRONTEND VALIDATION
  processWithdraw() validates:
    - amount >= 100
    - All fields filled

STEP 3 — API CALL
  POST /api/wallet/withdraw.php
  Body: { amount, bank_name, account_number, account_holder }

STEP 4 — BACKEND PROCESSING (Wallet::requestWithdrawal)
  1. Validate amount >= MIN_WITHDRAWAL (100)
  2. BEGIN TRANSACTION
  3. SELECT balance FOR UPDATE (row lock to prevent concurrent modifications)
  4. Check: amount <= balance (otherwise "Insufficient balance")
  5. UPDATE users: balance = balance - amount (DEDUCT immediately)
  6. INSERT INTO withdrawals (status='pending')
  7. INSERT INTO transactions (type='withdrawal', amount=NEGATIVE)
  8. COMMIT
  9. Return { success: true, withdrawal_id }

STEP 5 — ADMIN PROCESSES
  (Done via admin panel — admin/withdrawals.php)
  Admin reviews pending withdrawals
  Admin manually transfers money to user's bank
  Admin updates withdrawal status to 'approved' or 'rejected'
  If rejected: balance is refunded (separate admin action)

11. SECURITY MECHANISMS
================================================================================

1. SERVER-SIDE VERIFICATION
   Never trust frontend-provided amounts. complete_deposit.php ALWAYS calls
   Chapa's verify API to get the actual amount paid. The verified amount
   (from Chapa's response) is used for crediting, not the frontend amount.

2. ROW-LEVEL LOCKING (FOR UPDATE)
   All balance-modifying operations use SELECT ... FOR UPDATE to prevent
   race conditions when two requests try to process the same deposit
   simultaneously.

3. DATABASE TRANSACTIONS
   All critical operations (deposit completion, withdrawal) use 
   BEGIN TRANSACTION / COMMIT / ROLLBACK to ensure atomicity.

4. DOUBLE-CREDIT PREVENTION
   Before crediting, the system always checks if deposit.status is already
   'success'. If so, it returns the current balance without modifying it.

5. AUTHENTICATION
   All API endpoints (except chapa_callback.php) require active PHP session.
   requireLogin() sends 401 if not authenticated.

6. API KEY SECURITY
   Chapa's SECRET key is stored server-side only (constants.php).
   The PUBLIC key (CHAPUBK-...) is used in frontend JS for the inline SDK.
   These are DIFFERENT keys — the public key cannot be used to verify
   or initiate server-side transactions.

7. INPUT VALIDATION
   - Deposit amounts validated: >= MIN_DEPOSIT (5 ETB)
   - Withdrawal amounts validated: >= MIN_WITHDRAWAL (100 ETB)
   - Withdrawal checks: amount <= current balance
   - JSON input parsed safely with null coalescing (??)

8. PREPARED STATEMENTS
   All SQL queries use MySQLi prepared statements with parameter binding,
   preventing SQL injection.

9. IDEMPOTENT OPERATIONS
   createDepositRecord() checks for existing tx_ref before inserting.
   completeDepositDirect() checks for already-completed status.
   These ensure repeated calls don't cause duplicate records.

12. ERROR HANDLING & EDGE CASES
================================================================================

EDGE CASE 1: Chapa SDK fails to load
  → System detects via chapaSDKFailed flag
  → Falls back to redirect mode automatically
  → User sees "redirect mode" badge

EDGE CASE 2: Chapa's server is down (502, CORS error)
  → onPaymentFailure detects error keywords (502, cors, failed to fetch)
  → Shows error message with "Try Alternative Payment Method" button
  → Button triggers startRedirectPayment()

EDGE CASE 3: User closes browser during payment
  → Deposit record exists in DB as 'pending'
  → Chapa's callback URL is called server-to-server
  → handleChapaCallback() verifies and credits balance
  → User sees updated balance on next visit

EDGE CASE 4: Network error during complete_deposit.php
  → Frontend shows positive message ("Payment received!")
  → Page reloads after 5 seconds
  → If not yet credited, Chapa callback will handle it

EDGE CASE 5: Chapa verification delayed
  → complete_deposit.php returns { pending: true }
  → Frontend shows "Balance will update shortly"
  → Chapa callback eventually processes it

EDGE CASE 6: Double-click / duplicate requests
  → FOR UPDATE lock prevents concurrent processing
  → Status check (already 'success') prevents double credit
  → createDepositRecord() returns existing record if tx_ref exists

EDGE CASE 7: Amount mismatch (frontend says 100, user paid 50)
  → Server uses Chapa's verified amount, not frontend amount
  → $verifiedAmount = $chapaData['data']['amount']

EDGE CASE 8: Multiple payment methods (Telebirr, M-Pesa, CBE)
  → Phone validation logic detects carrier type:
     - 09xxxxxxxx = Telebirr
     - 07xxxxxxxx = M-Pesa (Safaricom)
  → Warns user if phone number doesn't match selected method
  → Prevents Chapa submit button until valid

EDGE CASE 9: Invalid email
  → Chapa.php falls back to 'customer@gmail.com'
  → Chapa requires a valid-looking email format

COMPLETE_DEPOSIT.PHP ERROR HANDLING:
  - Custom error handler converts PHP warnings to exceptions
  - Top-level try/catch wraps entire script
  - On any error: rollback, return error with debug info
  - HTTP 200 even on error (prevents browser error pages)
