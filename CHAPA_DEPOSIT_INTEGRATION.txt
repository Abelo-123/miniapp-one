1. OVERVIEW & ARCHITECTURE
================================================================================

The Chapa deposit system allows users to add funds (ETB - Ethiopian Birr) to
their Paxyo SMM panel account. It uses Chapa's INLINE CHECKOUT SDK embedded
directly into the app UI (not redirect-based).

HIGH-LEVEL FLOW:
┌─────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  User enters │───>│ Chapa Inline SDK │───>│ Chapa Payment    │
│  amount      │    │ renders in-app   │    │ Gateway processes│
└─────────────┘    └──────────────────┘    └──────────────────┘
                                                    │
                                                    ▼
┌─────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  Balance     │<───│ deposit_handler  │<───│ onSuccessful     │
│  updated     │    │ .php (backend)   │    │ Payment callback │
└─────────────┘    └──────────────────┘    └──────────────────┘

TWO CHECKOUT METHODS EXIST:
  1. INLINE CHECKOUT (Primary) — Used in smm.php via ChapaCheckout SDK
     - Renders payment form directly inside the app
     - Best UX, no page redirects
     - Requires domain whitelisting on Chapa dashboard

  2. REDIRECT CHECKOUT (Fallback) — Used in chapa_init.php & learn_deposit.php
     - Generates a Chapa checkout URL
     - Redirects user to Chapa's hosted payment page
     - Works on any domain (no CORS issues)

5. FRONTEND — JAVASCRIPT (Chapa SDK Integration)
================================================================================

Location: smm.php, inside the SMM object literal

─── initDeposit() ───
Called during app initialization. Sets up:

  1. AMOUNT CHIPS: Click handlers that ADD to current amount
     - Each chip has data-amount attribute (10, 100, 1000, 10000)
     - Clicking a chip adds its value to the input (accumulative)
     - Resets Chapa wrapper if it's already open (so new amount is used)
     - Visual feedback: briefly highlights the chip with accent color

  2. DEPOSIT BUTTON: Click handler
     - Validates minimum amount (>= 10 ETB)
     - Shows error toast if amount < 10
     - Calls this.startChapaPayment(amount)

  3. AMOUNT INPUT CHANGE: If user modifies the amount while Chapa is open
     - Hides the Chapa wrapper
     - Shows the deposit button again
     - Clears the Chapa inline form (forces re-initialization with new amount)


─── startChapaPayment(amount) ───
The core Chapa integration function. Steps:

  1. SDK CHECK: Verifies ChapaCheckout class is available
     - If not loaded yet, shows "Payment system loading..." toast
     - Retries after 1 second (recursive call)

  2. UI STATE: Hides deposit button, shows chapa-container-wrapper
     - Resets 'user-has-submitted' class
     - Hides error container
     - Clears previous form content

  3. PHONE VALIDATOR: Sets up after 1-second delay (waits for Chapa render)
     - Finds phone input inside Chapa's rendered form
     - Creates custom validation message element
     - Validates Ethiopian phone numbers in real-time
     (See Section 6 for full phone validation algorithm)

  4. CHAPA SDK INITIALIZATION:
     Configuration object passed to new ChapaCheckout():

     {
       publicKey: 'CHAPUBK-s9JQu74c7hAcdPPGxaAF6aT22Ih4HNtm',
       amount: Math.round(amount * 100) / 100,    // Rounded to 2 decimals
       currency: 'ETB',
       tx_ref: 'paxyo-' + userId + '-' + Date.now(),  // Unique per transaction
       title: 'Add Funds',
       description: 'Deposit to Paxyo',
       email: 'user-' + userId + '@paxyo.com',
       first_name: userName || 'Paxyo',
       last_name: 'User',
       showFlag: true,
       showPaymentMethodsNames: true,
       callbackUrl: window.location.origin + '/webhook_handler.php',
       onSuccessfulPayment: (result, refId) => { ... },
       onPaymentFailure: (error) => { ... },
       onClose: () => { ... }
     }

  5. CALLBACKS:

     onSuccessfulPayment(result, refId):
       - Logs success
       - Sets paymentSubmitted = true
       - Hides Chapa wrapper
       - Calls this.verifyDeposit(amount, refId)
       - refId fallback chain: refId || result?.tx_ref || 'ref-' + Date.now()

     onPaymentFailure(error):
       - SUPPRESSES phone/validation errors (false positives from SDK):
         Keywords suppressed: phone, mobile, valid, number, format,
         required, insert, enter
       - HANDLES network errors (close wrapper, show "Try Again"):
         Keywords: fetch, network, server, failed to fetch,
         connection, timeout
       - HANDLES regional errors:
         Keywords: no service found, country, not supported, region
       - SHOWS real failures:
         Keywords: cancel, decline, failed, reject
       - Unknown errors: logged but UI NOT disrupted

     onClose():
       - Hides loading
       - After 300ms: shows deposit button, hides Chapa wrapper
       - Resets 'user-has-submitted' class

  6. CLICK CAPTURE: Listens for clicks on buttons inside chapa-inline-form
     - Sets 'user-has-submitted' class on wrapper
     - Helps differentiate input errors from actual payment errors

  7. TRY/CATCH: Calls chapa.initialize()
     - On error: hides wrapper, shows deposit button
     - Special message for localhost: "Payment works only on production site"

6. FRONTEND — JAVASCRIPT (Phone Validation Algorithm)
================================================================================

Location: smm.php, inside startChapaPayment(), lines ~1652-1908

PURPOSE: Validates Ethiopian mobile phone numbers in real-time as the user
types into Chapa's phone input field. Prevents common mistakes and provides
clear guidance.

PAYMENT METHOD DETECTION:
  getSelectedPaymentMethod() function scans Chapa's rendered DOM to detect
  which payment method is active:
    - 'mpesa'    → M-Pesa / Safaricom (prefix: 07)
    - 'telebirr' → Telebirr / Ethio Telecom (prefix: 09)
    - 'bypass'   → CBE, E-Pay, E-Birr, Banks (no prefix restriction)

VALIDATION ALGORITHM — validatePhone(val, showMessages):

  INPUT PROCESSING:
    1. Remove all non-digit characters
    2. Strip international prefix '251' if present
    3. Detect and strip leading '0'
    4. Store base value (without leading zero)

  VALIDATION RULES:
    ┌─────────────────────────────────────────────────────────┐
    │ Step 1: PREFIX CHECK (skip for bypass methods)          │
    │   - If Telebirr selected and number starts with 7:     │
    │     ERROR: "This is an M-Pesa number"                   │
    │   - If M-Pesa selected and number starts with 9:       │
    │     ERROR: "This is a Telebirr number"                  │
    │   - If neither 7 nor 9: ERROR: "Must start with 09/07" │
    ├─────────────────────────────────────────────────────────┤
    │ Step 2: LENGTH CHECK                                    │
    │   - Less than 9 digits (base): HINT "Enter 9-10 digits"│
    │   - Exactly 9 digits: Valid if correct prefix           │
    │   - 10 digits starting with 0: Valid if correct prefix  │
    │   - More than 10 digits: ERROR "Number too long"        │
    └─────────────────────────────────────────────────────────┘

  VALID FORMATS:
    Telebirr: 9XXXXXXXX (9 digits) or 09XXXXXXXX (10 digits)
    M-Pesa:   7XXXXXXXX (9 digits) or 07XXXXXXXX (10 digits)
    Bypass:   Any 9-10 digit number

  UI FEEDBACK:
    - Success: Validation message hidden, Chapa Pay button ENABLED
    - Error:   Red message shown, Chapa Pay button DISABLED
    - Hint:    Message hidden (user still typing)

  EVENT LISTENERS:
    - input:  Real-time validation on each keystroke
    - blur:   Final validation when focus leaves input
    - focus:  Re-validate (payment method may have changed)
    - click:  Re-validate on ANY click in Chapa form
              (uses multiple timeouts: 10ms, 100ms, 300ms, 600ms
               to catch Chapa's slow DOM updates during method switches)

  BUTTON STATE MANAGEMENT:
    updateButtonState(enabled) — Finds Chapa's Pay Now button and:
      - Sets disabled property
      - Changes opacity (1 or 0.5)
      - Changes cursor (pointer or not-allowed)

─── verifyDeposit(amount, refId) ───

Called after Chapa's onSuccessfulPayment callback fires.

ALGORITHM:
  1. Send POST to deposit_handler.php:
     Body: { amount: <number>, reference_id: <string> }

  2. Parse JSON response

  3. IF data.status === 'success':
     a. Show success toast: "Deposit successful! Balance updated."
     b. Update balance locally: this.updateBalance(newBalance)
     c. Reset amount input to empty
     d. Update deposit balance display text
     e. Keep deposit button visible (user can deposit again)
     f. Hide Chapa wrapper
     g. Reload deposit history: this.loadDepositHistory()
     h. Show one-time guide: this.showDepositHistoryGuide()

  4. IF error:
     - Show error toast with message

  5. FINALLY:
     - Always hide loading indicator


─── updateBalance(newBalance) ───

Location: smm.php, lines ~1545-1554

  1. Check if balance actually changed (this.userBalance !== newBalance)
  2. Update internal state: this.userBalance = newBalance
  3. Update DOM element #user-balance with formatted ETB value
  4. Add 'pulse' CSS class for animation
  5. Remove 'pulse' class after 2 seconds

Purpose: FALLBACK method — generates a Chapa redirect checkout URL

REQUEST: POST with JSON body
  { "amount": <number>, "user_id": <string>, "user_name": <string> }

ALGORITHM:

  4. Generate unique tx_ref: 'paxyo-' + user_id + '-' + timestamp

       amount       = <user amount>

       first_name   = <user_name>
       last_name    = 'User'
       tx_ref       = <generated tx_ref>
       callback_url = http://localhost/paxyo/webhook_handler.php
       return_url   = http://localhost:5178/miniapp-one/?deposit_complete=1
                      &ref=<tx_ref>&amount=<amount>
       customization[title]       = 'Paxyo SMM Deposit'
       customization[description] = 'Add funds to your SMM panel account'
       public_key   = CHAPUBK-s9JQu74c7hAcdPPGxaAF6aT22Ih4HNtm

  6. Return JSON: { success: true, checkout_url: <url>, tx_ref: <ref> }

RESPONSE:
  Success: { "success": true, "checkout_url": "https://...", "tx_ref": "paxyo-..." }
  Error:   { "error": "Invalid amount" }

NOTE: This file uses LOCALHOST URLs and is primarily for development/fallback.
For production, URLs should be updated to the live domain.

12. BACKEND — deposit_handler.php (Deposit Processing & Balance Update)
================================================================================

Location: /paxyo/deposit_handler.php (81 lines)
Purpose: PRIMARY deposit processor — records deposit and updates user balance

REQUEST: POST with JSON body
  { "amount": <number>, "reference_id": <string> }

ALGORITHM:
  1. Include db.php and utils_bot.php
  2. Start session (with SameSite=None, Secure=true)
  3. Identify user:
     Priority: $_SESSION['tg_id'] → $_COOKIE['id'] → fallback to 111 (dev only)
  4. Validate: amount > 0 AND reference_id not empty
  5. DUPLICATE CHECK: Query deposits table for existing reference_id
     - If exists → return error: "Transaction already processed"
  6. INSERT deposit record:
     { user_id, amount, reference_id, status: 'completed' }
  7. UPDATE user balance:
     SQL: UPDATE auth SET balance = balance + {amount} WHERE tg_id = '{user_id}'
  8. GET new balance from database
  9. Return success response with new balance

  10. NOTIFY admin via notify_bot_admin():
      Data: { type: 'deposit', uid: user_id, uuid: first_name, amount: amount }

RESPONSE:
  Success: { "status": "success", "new_balance": <number>, "message": "Deposit successful" }
  Errors:
    { "status": "error", "message": "User not logged in" }
    { "status": "error", "message": "Invalid amount or reference ID" }
    { "status": "error", "message": "Transaction already processed" }

19. COMPLETE DEPOSIT FLOW (Step-by-Step)
================================================================================

STEP 1: USER NAVIGATES TO DEPOSIT TAB
  → Bottom nav "Deposit" button clicked
  → switchTab('deposit') called
  → deposit-view becomes visible
  → Balance display updated with current balance
  → loadDepositHistory() called to show recent deposits

STEP 2: USER ENTERS AMOUNT
  → Type in deposit-amount input field
  → OR click preset chips (+10, +100, +1k, +10k) to accumulate
  → If Chapa wrapper was open, it gets reset (hidden)

STEP 3: USER CLICKS "DEPOSIT" BUTTON
  → Validates amount >= 10 ETB
  → If invalid: shows error toast "Minimum deposit is 10 ETB"
  → If valid: calls startChapaPayment(amount)

STEP 4: CHAPA SDK INITIALIZATION
  → Checks if ChapaCheckout class is loaded
  → If not: waits 1 second and retries
  → Shows loading indicator
  → Hides deposit button, shows Chapa container
  → Creates new ChapaCheckout instance with configuration
  → Calls chapa.initialize() to render the payment form
  → Hides loading indicator

STEP 5: PHONE VALIDATION SETUP (after 1 second)
  → Finds Chapa's phone input in the rendered form
  → Creates custom validation message element
  → Attaches input/blur/focus/click event listeners
  → Initial validation triggered after 1.5 seconds

STEP 6: USER ENTERS PHONE & SELECTS PAYMENT METHOD
  → Real-time phone validation as user types
  → Payment method auto-detected from Chapa UI
  → Validation messages shown/hidden based on input
  → Pay button enabled/disabled based on validation

STEP 7: USER CLICKS "PAY NOW" (Chapa's button)
  → 'user-has-submitted' class added to wrapper
  → Chapa SDK processes the payment
  → User completes payment on their phone (USSD/STK push)

STEP 8A: PAYMENT SUCCESS
  → onSuccessfulPayment callback fires
  → Receives result object and refId
  → Hides Chapa wrapper
  → Calls verifyDeposit(amount, refId)

STEP 8B: PAYMENT FAILURE
  → onPaymentFailure callback fires
  → Error categorized:
    - Validation errors → suppressed (SDK false positives)
    - Network errors → show "Try Again" button
    - Regional errors → show service unavailable message
    - Real failures → show actual error message
    - Unknown errors → logged silently

STEP 9: BACKEND VERIFICATION (deposit_handler.php)
  → Receives POST: { amount, reference_id }
  → Identifies user from session/cookie
  → Checks for duplicate reference_id in deposits table
  → Inserts deposit record (status: 'completed')
  → Updates auth.balance += amount
  → Gets new balance
  → Notifies admin via log file
  → Returns: { status: 'success', new_balance: X }

STEP 10: FRONTEND UPDATES
  → Success toast shown
  → Balance display updated (with pulse animation)
  → Amount input cleared
  → Deposit button shown again (for repeat deposits)
  → Deposit history reloaded
  → One-time guide shown (scroll to history, highlight)

STEP 11: USER CAN DEPOSIT AGAIN
  → No redirect to other tabs
  → Deposit button remains visible
  → User can enter new amount and repeat process

20. ERROR HANDLING STRATEGY
================================================================================

FRONTEND ERROR CATEGORIES:

  1. SDK NOT LOADED
     → Toast: "Payment system loading... please wait"
     → Auto-retry after 1 second

  2. CHAPA VALIDATION ERRORS (suppressed)
     → Keywords: phone, mobile, valid, number, format, required, insert, enter
     → Action: Silently ignored (console.log only)
     → Reason: Chapa SDK fires these incorrectly during valid input

  3. NETWORK ERRORS
     → Keywords: fetch, network, server, failed to fetch, connection, timeout
     → Action: Hide wrapper, show "Try Again" button
     → Toast: "Connection failed. Please check your internet"

  4. REGIONAL ERRORS
     → Keywords: no service found, country, not supported, region
     → Action: Hide wrapper, show "Try Again" button
     → Toast: "Payment service unavailable in your region"

  5. REAL PAYMENT FAILURES
     → Keywords: cancel, decline, failed, reject
     → Action: Hide wrapper, show "Try Again" button
     → Toast: Actual error message or generic "Payment failed"

  6. UNKNOWN ERRORS
     → Action: Console log only, no UI disruption

  7. CHAPA INITIALIZATION ERROR
     → Localhost: "Payment works only on production site (paxyo.com)"
     → Production: "Could not initialize payment. Please try again."

  8. PHONE VALIDATION ERRORS (custom)
     → Wrong prefix for selected method
     → Number too short/long
     → Visual: Red validation message above Pay button

CSS ERROR SUPPRESSION:
  → All native Chapa error elements are hidden via CSS !important rules
  → Custom validation handled entirely by JavaScript

BACKEND ERROR HANDLING:
  → User not logged in → error response
  → Invalid amount or reference → error response
  → Duplicate transaction → error response (prevents double-counting)
  → Database error → error response

22. KNOWN ISSUES & WORKAROUNDS
================================================================================

1. CHAPA SDK FALSE POSITIVE ERRORS
   Issue:   SDK fires onPaymentFailure with phone validation errors even when
            the phone number is valid and payment is processing
   Workaround: Frontend suppresses all validation-keyword errors in the
               onPaymentFailure callback + CSS hides all native error elements

2. CHAPA CORS ON LOCALHOST
   Issue:   Chapa inline checkout requires domain whitelisting
   Workaround: Show message "Payment works only on production site (paxyo.com)"
               when running on localhost
   Solution: Use redirect checkout (chapa_init.php) for development

3. DELAYED CHAPA FORM RENDERING
   Issue:   Chapa's form takes time to render, phone input not immediately available
   Workaround: setTimeout of 1000ms before attempting to find phone input
               Multiple staggered timeouts (10, 100, 300, 600ms) for click re-validation

4. PAYMENT METHOD SWITCH RE-VALIDATION
   Issue:   When user switches payment method, phone may become invalid for new method
   Workaround: Click event listener on entire chapaInline form with multiple
               staggered timeouts to catch DOM updates

5. SESSION LOCK BLOCKING
   Issue:   PHP sessions block concurrent requests (SSE streams, AJAX)
   Workaround: session_write_close() called immediately after reading user_id
               in get_deposits.php
